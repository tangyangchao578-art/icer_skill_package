# 功能安全设计原则

## 目的

定义符合 ISO 26262 标准的功能安全设计原则，确保安全相关故障被正确识别、控制和缓解。

## 基本概念

### ASIL 等级

- **QM**：质量管理，不要求安全机制
- **ASIL A**：低风险
- **ASIL B**：中低风险
- **ASIL C**：中高风险
- **ASIL D**：最高风险

安全要求随 ASIL 等级提升而更加严格。

### 安全目标

- 识别潜在危险
- 定义安全目标
- 分解为安全需求
- 逐层分配到设计模块

## 安全机制

### 常用安全机制

#### 单点故障检测：
- [ ] 双锁双核（Dual Core Lockstep）
- [ ] 信息冗余（编码保护）
- [ ] 奇偶校验
- [ ] ECC 错误校正
- [ ] 看门狗
- [ ] 窗口看门狗
- [ ] 超时检测

#### 瞬态故障缓解：
- [ ] 刷新机制
- [ ] 重试机制
- [ ] 错误纠正（ECC）

#### 安全关机：
- [ ] 安全状态机
- [ ] 独立的安全监控
- [ ] 故障响应进入安全状态

### 自由度分解

- ASIL D 要求：使用独立的冗余通道
- 不同 ASIL 等级的功能在设计上隔离
- 避免共因故障

## 设计流程

### 要求跟踪

- [ ] 从安全目标到具体实现的可跟踪性
- [ ] 每个安全需求都有对应的实现
- [ ] 每个安全需求都有对应的验证

### FMEDA（失效模式影响和诊断分析）

- 识别每个模块的潜在失效模式
- 分析对系统的影响
- 分配诊断覆盖率
- 验证达到要求的覆盖率目标

### 安全分析

- [ ] 单点故障度量（SPFM）满足要求
- [ ] 潜在故障度量（LFM）满足要求
- [ ] 随机硬件故障符合 ASIL 要求
- [ ] 系统故障分析完成

## 实现准则

- [ ] 安全机制独立于功能逻辑
- [ ] 安全机制不能被正常功能屏蔽
- [ ] 故障检测后必须进入安全状态
- [ ] 错误信号必须正确连接到错误处理模块
- [ ] 安全相关配置寄存器受保护
- [ ] 安全机制有可测试性设计

## 验证要求

- [ ] 故障注入测试验证安全机制有效性
- [ ] 每个安全机制都必须被验证
- [ ] 诊断覆盖率测量满足要求
- [ ] 安全时间间隔满足要求
- [ ] 独立于原设计的验证环境

## 文档要求

- [ ] 安全需求规格
- [ ] 安全架构文档
- [ ] FMEDA 报告
- [ ] 安全验证报告
- [ ] 可追踪性矩阵

- [ ] 诊断覆盖率满足要求
- [ ] 可跟踪性完整
- [ ] 验证完成
