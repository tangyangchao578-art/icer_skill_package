# 功能安全分析技能

当用户需要进行功能安全分析和安全机制设计时，启用此技能。符合 ISO 26262 标准，支持 ASIL A-D 各个等级。

## When to Activate

- 项目开始安全需求分解
- 模块级 FMEDA 分析
- 安全机制设计
- 故障注入测试规划
- 功能安全验证规划
- 文档整理符合 ISO 26262

---

## 步骤 1：确定 ASIL 等级

### 1.1 理解 ASIL 定义

ASIL (Automotive Safety Integrity Level) 是 ISO 26262 定义的功能安全等级。

| ASIL 等级 | 安全要求 | SPFM 最低 | LFM 最低 | 典型应用 |
|-----------|----------|-----------|----------|----------|
| **QM** | 无安全要求 | - | - | 娱乐系统 |
| **ASIL-A** | 基本 | ≥ 90% | ≥ 60% | 后视摄像头 |
| **ASIL-B** | 标准 | ≥ 90% | ≥ 60% | 仪表盘 |
| **ASIL-C** | 较高 | ≥ 97% | ≥ 80% | 转向辅助 |
| **ASIL-D** | 最高 | ≥ 99% | ≥ 90% | 刹车系统 |

### 1.2 进行危害分析和风险评估（HARA）

**做什么：** 识别潜在危害，确定 ASIL 等级。

**步骤：**
1. 列出所有潜在危害
2. 评估严重度（S1/S2/S3）
3. 评估暴露率（E1/E2/E3/E4）
4. 评估可控性（C1/C2/C3）
5. 查 ISO 26262 表格确定 ASIL

**输出：** 危害分析表

| 危害 ID | 危害描述 | 严重度 | 暴露率 | 可控性 | ASIL |
|---------|----------|--------|--------|--------|------|
| H001 | 刹车信号丢失 | S3 | E4 | C3 | ASIL-D |
| H002 | 转向信号错误 | S3 | E3 | C3 | ASIL-C |
| H003 | 仪表盘显示错误 | S2 | E3 | C2 | ASIL-A |

### 1.3 定义安全目标

**做什么：** 为每个危害定义安全目标。

| 危害 ID | 安全目标 | 故障容错时间 |
|---------|----------|--------------|
| H001 | 刹车信号必须正确 | 100ms |
| H002 | 转向信号必须正确 | 50ms |

## 步骤 2：识别组件和失效模式

### 2.1 列出所有硬件模块

**做什么：** 列出设计中所有硬件模块。

| 模块 ID | 模块名称 | 描述 |
|---------|----------|------|
| M001 | CPU | 处理器内核 |
| M002 | SRAM | 数据存储器 |
| M003 | Flash | 程序存储器 |
| M004 | CAN 控制器 | 通信接口 |
| M005 | ADC | 模数转换器 |

### 2.2 识别每个模块的失效模式

**常见失效模式：**

| 失效模式 | 描述 |
|----------|------|
| stuck-at-0 | 信号固定为 0 |
| stuck-at-1 | 信号固定为 1 |
| bit flip | 比特翻转（瞬态） |
| 数据损坏 | 数据错误 |
| 时序错误 | 建立保持失败 |
| 开路 | 连接断开 |
| 短路 | 连接短路 |

**输出：** 失效模式表

| 模块 | 失效模式 | 失效率 (FIT) |
|------|----------|--------------|
| CPU | 永久故障 | 100 FIT |
| CPU | 瞬态故障 | 500 FIT |
| SRAM | 单比特翻转 | 1000 FIT |
| SRAM | 多比特翻转 | 10 FIT |
| Flash | 数据损坏 | 50 FIT |

## 步骤 3：分析失效影响

### 3.1 分析对安全目标的影响

**做什么：** 分析每个失效模式是否违反安全目标。

| 模块 | 失效模式 | 影响 | 是否违反安全目标 |
|------|----------|------|------------------|
| CPU | 永久故障 | 系统死机 | 是（SG_1） |
| SRAM | 单比特翻转 | 数据错误 | 可能（SG_2） |
| Flash | 数据损坏 | 程序错误 | 是（SG_1） |

### 3.2 影响分级

| 分级 | 描述 |
|------|------|
| 无影响 | 不影响功能，不影响安全 |
| 轻微影响 | 影响非安全功能 |
| 严重影响 | 影响安全功能 |
| 危险影响 | 违反安全目标 |

---

## 步骤 4：设计安全机制

### 4.1 选择安全机制

**做什么：** 为每个违反安全目标的失效模式分配安全机制。

| 模块 | 失效模式 | 安全机制 | 诊断覆盖率 |
|------|----------|----------|------------|
| CPU | 永久故障 | 双核锁步 + 看门狗 | 99% |
| CPU | 瞬态故障 | ECC | 99% |
| SRAM | 单比特翻转 | ECC (SECDED) | 99% |
| SRAM | 多比特翻转 | 双拷贝比较 | 90% |
| Flash | 数据损坏 | ECC + CRC | 99% |

### 4.2 安全机制详细设计

#### CPU 安全机制

| 安全机制 | 针对失效 | 诊断覆盖率 | 实现复杂度 |
|----------|----------|------------|------------|
| 双核锁步 | 永久/瞬态 | 99% | 高 |
| 看门狗 | 永久 | 90% | 低 |
| 内存保护 | 软件错误 | 60% | 中 |
| 奇偶校验 | 瞬态 | 90% | 低 |

#### 存储器安全机制

| 安全机制 | 针对失效 | 诊断覆盖率 | 纠正能力 |
|----------|----------|------------|----------|
| ECC (SECDED) | 单比特错误 | 99% | 可纠正 |
| 双拷贝 | 多比特错误 | 90% | 可检测 |
| CRC | 数据完整性 | 99% | 可检测 |
| BIST | 固定故障 | 90% | 可检测 |

## 步骤 5：计算安全指标

### 5.1 计算 SPFM

**公式：**
```
SPFM = 1 - (单点故障失效率 / 总故障失效率)
```

**计算示例：**

| 失效模式 | 失效率 (FIT) | 安全机制 | DC | 残留失效率 |
|----------|--------------|----------|-----|------------|
| CPU 永久 | 100 | 锁步 | 99% | 1 |
| CPU 瞬态 | 500 | ECC | 99% | 5 |
| SRAM 单比特 | 1000 | ECC | 99% | 10 |
| 总计 | 1600 | | | 16 |

```
SPFM = 1 - (16 / 1600) = 99%
```

### 5.2 计算 LFM

**公式：**
```
LFM = 1 - (潜伏故障失效率 / 总故障失效率)
```

### 5.3 检查是否满足 ASIL 要求

| ASIL | SPFM 要求 | LFM 要求 | 实际 SPFM | 实际 LFM | 是否满足 |
|------|-----------|----------|-----------|----------|----------|
| ASIL-D | ≥ 99% | ≥ 90% | 99% | 92% | 满足 |

## 步骤 6：避免共因故障

### 6.1 识别共因故障来源

| 来源 | 风险 |
|------|------|
| 共用电源 | 电源故障同时影响冗余模块 |
| 共用时钟 | 时钟故障同时影响冗余模块 |
| 共用软件 | 软件 bug 同时影响冗余模块 |
| 环境因素 | 温度、辐射同时影响 |

### 6.2 避免措施

| 共因因素 | 避免措施 |
|----------|----------|
| 共用电源 | 独立电源域 |
| 共用时钟 | 独立时钟源 |
| 共用软件 | 多样化软件实现 |
| 温度 | 温度监控 |
| 辐射 | 屏蔽、ECC |

## 步骤 7：功能安全验证

### 7.1 故障注入测试

**做什么：** 注入故障，验证安全机制是否正确检测和响应。

**测试步骤：**
1. 注入单点故障
2. 检查安全机制是否检测到
3. 检查是否进入安全状态
4. 统计诊断覆盖率

### 7.2 安全状态验证

**检查清单：**
- [ ] 故障检测后正确进入安全状态
- [ ] 安全状态输出值正确
- [ ] 安全状态持续时间满足要求
- [ ] 故障清除后正确恢复

### 7.3 安全指标验证

**检查清单：**
- [ ] SPFM 满足 ASIL 要求
- [ ] LFM 满足 ASIL 要求
- [ ] 所有安全机制都已验证
- [ ] 故障注入测试通过

## 反模式（要避免）

❌ **安全机制设计不足**：SPFM/LFM 不满足 ASIL 要求

❌ **忽视共因故障**：冗余系统同时失效

❌ **安全状态设计不当**：故障后无法进入安全状态

❌ **测试不充分**：没有故障注入测试

❌ **可追踪性不全**：某个安全需求找不到实现或验证

❌ **安全机制被功能故障屏蔽**：功能和安全机制同时失效
