# 功能安全分析技能

当用户需要进行功能安全分析和安全机制设计时，启用此技能。符合 ISO 26262 标准，支持 ASIL A-D 各个等级。

## When to Activate

- 项目开始安全需求分解
- 模块级 FMEDA 分析
- 安全机制设计
- 故障注入测试规划
- 功能安全验证规划
- 文档整理符合 ISO 26262

## ISO 26262 基本概念

### ASIL 等级（汽车安全完整性等级）

| 等级 | 风险程度 | 要求 | SPFM 最低 | LFM 最低 |
|------|----------|------|-----------|-----------|
| **QM** | 质量管理 | 无安全要求 | - | - |
| **ASIL A** | 低风险 | 基本 | ≥ 90% | ≥ 60% |
| **ASIL B** | 中低风险 | 标准 | ≥ 90% | ≥ 60% |
| **ASIL C** | 中高风险 | 较高 | ≥ 97% | ≥ 80% |
| **ASIL D** | 最高风险 | 最高 | ≥ 99% | ≥ 90% |

ASIL 等级越高，要求越严格。

### 安全生命周期

按照 ISO 26262 流程：

1. **概念阶段** → 安全目标定义
2. **系统级开发** → 安全需求分配
3. **硬件级开发** → 安全机制实现
4. **软件级开发** → 安全软件实现
5. **集成验证** → 验证所有安全需求
6. **功能安全评估** → 独立评估
7. **生产发布**

## FMEDA 分析流程

**F**ailure **M**odes, **E**ffects, and **D**iagnostic **A**nalysis

### 完整步骤

1. 列出设计中所有硬件模块
2. 识别每个模块的潜在失效模式
3. 分析每个失效模式对上层模块的影响
4. 分析对安全目标的影响
5. 是否违反安全目标？
6. 分配对应的安全机制
7. 分配诊断覆盖率（DC）
8. 计算 SPFM 和 LFM 度量
9. 检查满足 ASIL 要求

### 常见失效模式

| 失效模式 | 描述 |
|----------|------|
| stuck-at-0 | 信号固定永远 0 |
| stuck-at-1 | 信号固定永远 1 |
| transition error | 翻转错误，延迟错误 |
| timing error | 时序错误，建立保持失败 |
| bit flip | 存储单元翻转错误 |
| open | 连接开路 |
| short | 连接短路 |

### 影响分级

- **无影响** → 不影响功能，不影响安全
- **轻微影响** → 影响非安全功能，不影响安全
- **严重影响** → 影响安全功能
- **危险影响** → 违反安全目标

只有违反安全目标的单点故障才需要安全机制。

## 常用安全机制

### 单点故障检测

| 安全机制 | 适用场景 | 诊断覆盖率 (DC) |
|----------|----------|------------------|
| 奇偶校验 | 存储器，总线 | 低 ~ 中（~50-70%） |
| ECC | 存储器 | 高（> 99%） |
| 双核锁步 | 处理器内核，控制逻辑 | 高（> 99%） |
| 简单看门狗 | 程序运行监控 | 中 ~ 高（~80-90%） |
| 窗口看门狗 | 时钟频率监控 | 高（> 99%） |
| 双模冗余比较 | 关键控制逻辑 | 高（> 99%） |
| 时间冗余重试 | 计算 | 中（~70-80%） |
| ECC 纠错 | 存储器瞬态故障 | 高 |

### 容错机制（检测到故障后怎么办）

- **ECC 纠错**：自动纠正单比特错误
- **冗余模块切换**：故障模块切换到冗余模块
- **重试**：瞬态故障重试计算
- **安全关机**：进入安全状态，输出安全值
- **报警**：通知软件处理故障

## 避免共因故障

### 什么是共因故障

一个共同原因导致两个冗余模块同时故障 → 两个都错了，检测不到。

### 避免方法

- 空间分离：两个模块物理位置分开
- 独立时钟：两个模块使用不同时钟源
- 独立复位：两个模块使用不同复位
- 独立电源：两个模块使用不同电源域
- 不同设计：两个模块不同设计团队/不同方法

### 独立性

ASIL D 要求安全机制和功能逻辑独立，不能被同一个故障同时破坏。

## 度量计算

### SPFM（Single-Point Fault Metric）

公式：
```
SPFM = ( 1 - ( Σ( λ_SP * (1 - DC) ) / Σ λ_SP ) ) * 100%
```

- λ_SP = 单点故障失效率
- DC = 诊断覆盖率

### LFM（Latent Fault Metric）

公式：
```
LFM = ( 1 - ( Σ λ_LP / Σ λ_total ) ) * 100%
```

- λ_LP = 潜在故障失效率（未被检测到的残留故障）

## 设计准则

✅ **必须遵守：**

1. **安全机制独立**：安全机制独立于功能逻辑
2. **单点故障不违反安全目标**：任何单点故障不能导致违反安全目标
3. **安全机制不能被功能故障屏蔽**：故障发生了，安全机制必须能检测到
4. **故障检测后必须进入安全状态**：检测到故障，立即进入安全状态
5. **可测试性**：安全机制必须可以测试

## 验证要求

### 必须验证：

- [ ] 每个安全机制都能正确检测对应故障
- [ ] 故障注入测试，每个故障都能被检测
- [ ] 故障检测后，正确进入安全状态
- [ ] 诊断覆盖率测量满足要求
- [ ] SPFM 和 LFM 满足 ASIL 要求
- [ ] 所有安全需求都有验证证据

### 故障注入测试

- 注入每种失效模式
- 检查安全机制是否检测到
- 检查安全响应是否正确
- 统计诊断覆盖率

## 可追踪性要求

必须有可追踪性矩阵：

| 安全目标 | 安全需求 | 实现模块 | 验证测试 | 结论 |
|----------|----------|----------|----------|------|
| SG_1 | SR_1 | module | test_1 | pass |

从安全目标 → 需求 → 实现 → 验证，每一环都可追踪。

## 文档要求

功能安全项目必须输出这些文档：

- [ ] 安全需求规格说明书
- [ ] 安全架构设计文档
- [ ] FMEDA 分析报告（Excel 表格）
- [ ] 安全验证报告（含故障注入结果）
- [ ] 可追踪性矩阵
- [ ] 功能安全评估报告

## 检查清单

FMEDA 完成后检查：

- [ ] 所有模块都已识别
- [ ] 所有失效模式都已识别
- [ ] 所有影响安全的单点故障都有安全机制
- [ ] 诊断覆盖率正确分配
- [ ] SPFM 满足 ASIL 要求
- [ ] LFM 满足 ASIL 要求
- [ ] 共因故障已经考虑
- [ ] 所有安全需求都可追踪
- [ ] 验证计划覆盖所有安全机制

## 常见错误

❌ **忽略共因故障**：两个冗余模块，同一个原因一起坏 → 检测不到 → 不安全

❌ **诊断覆盖率高估**：实际达不到，度量不满足 ASIL 要求

❌ **可追踪性不全**：某个安全需求找不到实现，找不到验证 → 审核不通过

❌ **安全机制被功能故障屏蔽**：功能坏了，安全机制也坏了，检测不到 → 不安全

✅ **每个安全需求都追踪到实现和验证**
