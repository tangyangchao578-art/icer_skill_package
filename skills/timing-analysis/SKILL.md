# 静态时序分析技能

当用户需要进行静态时序分析和时序收敛时，启用此技能。提供时序结果解读、违例调试、优化策略。

## When to Activate

- 静态时序分析结果解读
- 时序违例调试
- 时序收敛优化
- 多工艺角时序分析
- OCV 分析

## 基本概念

### 建立时间（Setup Time）

数据需要在下一个时钟沿之前稳定到达输入端：

**要求：**
```
data_arrival_time - clock_arrival_time <= required_time
```
**Slack = required_time - (data_arrival_time - clock_arrival_time)**

- Slack >= 0 → 满足
- Slack < 0 → 违例，路径太慢
- 越大的负 slack 越严重

### 保持时间（Hold Time）

数据需要在时钟沿之后保持稳定，不能太快到达：

**要求：**
```
data_arrival_time - previous_data_arrival_time >= required_time
```
**Slack = (data_arrival_time - previous_data_arrival_time) - required_time**

- Slack >= 0 → 满足
- Slack < 0 → 违例，路径太快

### 关键术语

| 术语 | 含义 |
|------|------|
| **WNS** | Worst Negative Slack → 最差的负 slack，WNS >= 0 才收敛 |
| **TNS** | Total Negative Slack → 所有负 slack 总和，反映整体情况 |
| **NP** | Number of Paths → 违例路径数量 |
| **OCV** | On-Chip Variation → 片上变化 |
| **CPPR** | Clock Path Pessimism Removal → 去除时钟路径悲观 |

### 时钟不确定性

- 预留余量给时钟抖动
- 预留余量给 OCV（片上变化）
- 先进工艺需要更大余量：
  - 28nm 以上：~5% 周期
  - 16nm 以下：~10% 周期

### 延迟计算

- **单元延迟**：基于输入斜率和输出负载查表（非线性）
- **线延迟**：基于线长、线宽、间距计算 RC，然后计算延迟
- 先进工艺线延迟占总延迟 60-70%，比单元延迟更大

## 静态时序分析完整流程

```
1. 读入网表
   ↓
2. 读入工艺库 (.lib)
   ↓
3. 读入时序约束 (.sdc)
   ↓
4. 构建时序图（单元 + 网 → 时序节点）
   ↓
5. 计算每个节点到达时间
   ↓
6. 检查建立时间和保持时间
   ↓
7. 生成时序报告（WNS, TNS, 违例列表）
```

## 时序报告解读

### 第一步：看什么

1. **看所有工艺角** → 每个工艺角都要满足
2. **看 WNS** → 最差负 slack 必须 >= 0
3. **看 TNS** → 总负 slack 越小越好
4. **看违例数量** → 越少越好

### 第二步：分析最差路径

拿到最差路径，分析：

```
起点  →  组合逻辑  →  ...  → 终点
  单元延迟 XX%
  线延迟   XX%
```

问题在哪？
- 如果线延迟占比太大 → 问题在布局，重新摆放
- 如果单元延迟占比太大 → 问题在逻辑结构，分割路径

### 第三步：分类违例

- 都是同一个模块的路径 → 优化这个模块
- 都是长路径 → 流水线分割
- 随机分布 → 整体优化

## 优化策略

### 建立时间违例优化（路径太慢）

按效果排序：

| 方法 | 效果 | 适用场景 |
|------|------|----------|
| 1. 逻辑重构分割长组合路径 | 🟢 最大 | 长组合逻辑路径 |
| 2. 换更大驱动单元 | 🟡 中等 | 负载大，驱动不够 |
| 3. 复制高扇出驱动 | 🟡 中等 | 一个信号驱动很多负载 |
| 4. 重新摆放模块 | 🟡 中等 | 关键路径线太长 |
| 5. 调整有用时钟偏斜 | ⚪ 小 | 接近收敛最后调整 |
| 6. Retiming 移动寄存器 | ⚪ 小 | 工具支持 |

**示例：长组合逻辑分割**

```
❌ 不好：一级组合逻辑 32 级进位，延迟太大
always @(*) begin
  for (i=0; i<32; i=i+1) begin
    c[i+1] = a[i] & b[i] | c[i] & (a[i]^b[i]);
  end
end

✅ 好：分成两级流水
always @(posedge clk) begin
  level1_c <= lower 16 bits calculation;
end
always @(*) begin
  final result = upper 16 bits + level1_c;
end
```

### 保持时间违例优化（路径太快）

| 方法 | 效果 | 适用场景 |
|------|------|----------|
| 1. 在快路径插入延迟单元 | 🟢 最大 | 单个路径违例 |
| 2. 调整有用时钟偏斜 | 🟡 中等 | 多个路径在同一个时钟域 |
| 3. 换成更小驱动 | 🟡 中等 | 驱动太强太快 |
| 4. 重新摆放增加线长 | ⚪ 小 | 布局空间足够 |

**⚠️ 重要：** 修复保持时间后必须**重新检查建立时间**。修复保持时间可能会影响建立时间。

## 多工艺角分析

必须检查所有这些工艺角：

| 工艺角 | 晶体管速度 | 电压 | 温度 | 最容易出现 |
|--------|------------|------|------|------------|
| **FF** | 快-快 | 高 | 低 | 保持时间违例 |
| **SS** | 慢-慢 | 低 | 高 | 建立时间违例 |
| **TT** | 典型 | 典型 | 典型 | 参考 |
| **FS** | 快N慢P | 典型 | 典型 | 混合偏差 |
| **SF** | 慢N快P | 典型 | 典型 | 混合偏差 |

**要求：** **所有工艺角都必须满足时序要求**，才能算收敛。

## OCV（片上变化）

### 什么是 OCV

制造工艺偏差导致：同一个芯片上，同一个类型的两个单元速度也可能不同。

- OCV 分析考虑这种偏差
- 计算最坏情况偏差
- 增加额外时序余量
- 先进工艺（16nm 以下）必须做 OCV
- 保持时间对 OCV 更敏感，偏差更容易导致保持违例

### AOCV vs POCV

- **AOCV** (Advanced OCV)：基于距离偏差，越远偏差越大
- **POCV** (Probabalistic OCV)：基于统计，更准确
- 先进工艺推荐 POCV

## 为什么保持时间修复比建立时间更难？

1. **阶段问题**：保持修复在布局布线之后，已经不能大规模改变布局
2. **方向问题**：保持违例需要**增加**延迟，建立违例需要**减少**延迟。增加延迟选择少
3. **影响范围**：改变一条路径延迟可能影响其他路径保持时间，容易引入新违例
4. 需要多次迭代，每次迭代一点一点修复

## 大量违例调试步骤

当有上百个违例，从哪里开始？

1. **排序**：按 slack 从小到大排序，从最差的开始
2. **分类**：同一个模块的违例放一起，先修复最差模块
3. **修复一个，重新运行 STA** → 看修复效果，有没有引入新问题
4. **逐步迭代**：一次不要修复太多，容易引入新问题
5. **收敛**：直到 WNS >= 0，所有违例修复

## 常见错误

### ❌ 只检查典型工艺角

只检查 TT，不检查 FF/SS → 流片后发现时序不满足，芯片不能工作。

### ✅ 必须检查所有工艺角。

### ❌ 忽略 CPPR

不打开 CPPR → 时钟路径悲观，WNS 比实际差，过度优化。

### ✅ 打开 CPPR 去除时钟路径悲观，得到更准确结果。

### ❌ 保持时间修复完不重新检查建立

修复保持增加了延迟，可能导致建立时间违例。

### ✅ 修复完成后，全流程重新检查建立时间。

## 检查清单

时序收敛完成检查：

- [ ] 所有工艺角 WNS ≥ 0
- [ ] 所有工艺角 TNS = 0（零违例）
- [ ] 所有保持时间满足
- [ ] OCV 分析满足要求
- [ ] CPPR 已经打开，结果准确
- [ ] 没有未检查的 false path
